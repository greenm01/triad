# Data-Oriented Design (DOD) Architecture in Triad

This document outlines the Data-Oriented Design (DOD) architecture principles adopted by the `triad` window manager.

The architectural pattern described here is heavily inspired by Yehonathan Shavrit's "Data Oriented Design" concepts and directly mirrors the successful DOD implementation found in the `ec4x` game engine (`dev/ec4x/src/engine/types/`).

## Core Philosophy: Strict Separation of Code and Data

In true DOD fashion, `triad` strictly separates the structural definition of state (Data) from the logic that transforms it (Code/Systems).

We do not use Object-Oriented paradigms where "Windows" own their methods or "Tags" own lists of Windows. Instead, the application state is a centralized database of flat collections, and logic operates as pure functions or "Systems" that iterate over these collections.

This approach yields three primary benefits:
1. **Cache Locality:** Flat arrays are extremely CPU cache-friendly.
2. **Simplified Relationships:** Many-to-many relationships (like `dwm`-style tags) become trivial relational joins rather than complex nested object trees.
3. **Type Safety:** We can leverage the compiler to enforce correctness without incurring runtime overhead.

---

## 1. Distinct IDs

Every logical entity in `triad` is represented by a strictly typed, zero-overhead identifier.

Instead of passing generic integers or Wayland surface pointers around, we define distinct ID types. This prevents subtle bugs, such as accidentally passing a `TagId` to a function expecting a `WindowId`.

```nim
type
  WindowId* = distinct uint32
  TagId*    = distinct uint32
  ColumnId* = distinct uint32
  OutputId* = distinct uint32
```
*Note: ID `0` is universally reserved to represent a "Null" or "None" state.*

## 2. Centralized, Monotonic ID Generation

IDs are not tied to the lifecycle of the underlying display server (e.g., Wayland Surface IDs, which can be recycled). They are logical, monotonically increasing integers generated by a centralized counter in the application state.

```nim
type
  IdCounters* = object
    nextWindowId*: uint32
    nextTagId*: uint32
    nextColumnId*: uint32

# ID Generators guarantee uniqueness and never issue '0'
proc generateWindowId*(model: var Model): WindowId = ...
```

## 3. The `EntityManager` Pattern (Swap-and-Pop)

Entities and their associated data (Components) are not stored in nested graphs or linked lists. They are stored in an `EntityManager` collection.

The `EntityManager` consists of two parts:
1. A flat `seq[T]` (the actual data array), ensuring dense, contiguous memory.
2. A `Table[ID, int]` index, mapping the logical ID to its current physical index in the array.

```nim
type
  EntityManager*[ID, T] = object
    data*: seq[T]
    index*: Table[ID, int]
```

### Constant-Time $O(1)$ Deletion
When a window closes, we must remove it from the array without leaving "holes" or incurring the cost of shifting the entire array down. `triad` uses the "swap-and-pop" technique:
1. Move the *last* element in the `data` array into the slot of the deleted element.
2. Update the `index` table for the moved element.
3. Pop the last element off the array.

## 4. Flattening the Tag Hierarchy

Because `triad` uses a `dwm`-style tag system (where a window can exist on multiple tags simultaneously), nesting data structures (e.g., `Tag -> seq[Column] -> seq[Window]`) creates synchronization nightmares.

Under the DOD architecture, relationships are modeled as external relational tables rather than object ownership.

```nim
type
  Model* = object
    # Core Data (The "Entities")
    windows*: EntityManager[WindowId, WindowData]
    tags*:    EntityManager[TagId, TagState]
    columns*: EntityManager[ColumnId, ColumnData]
    outputs*: EntityManager[OutputId, OutputData]

    # Relationships (The "Components")
    # A window's presence on a tag is just a relational mapping.
    windowTags*: Table[WindowId, uint32] # uint32 bitmask of TagIds
    windowColumns*: Table[WindowId, ColumnId]
```

To determine which windows are on a specific tag, a layout system queries the `windowTags` relationship, grabs the IDs, and retrieves the highly-localized data directly from the `windows.data` array.
